GS QA E2E Testing - Project Overview
====================================

1) What this project is
-----------------------
This is a TypeScript BDD automation framework using:
- Cucumber for behavior-driven test execution
- Playwright for browser UI automation and API request testing
- Cucumber HTML Reporter for human-readable run reports

It validates a Good Sam / Camping World ESP checkout platform via both UI and API scenarios.

2) Main capabilities
--------------------
A) UI E2E Checkout Automation
- Launches checkout app
- Verifies checkout page UI elements
- Completes personal details
- Completes vehicle details for Auto and RV flows
- Customizes plan and secures quote
- Verifies payment page
- Enters payment details in iframe

B) API Automation
- Address validation API tests (positive + negative validations)
- Full vehicle hierarchy extraction by year (autos/rvs):
  years -> manufacturers -> models -> trims -> styles/vehicleId
- Exports results and failures to CSV files in output/

3) High-level structure
-----------------------
- features/
  - api/
    - address-validate.feature
    - vehicle-year-full-extract.feature
  - ui/
    - checkout-e2e.feature

- src/
  - hooks/ (test lifecycle setup/teardown)
  - step-definitions/ (Given/When/Then implementations)
  - ui/
    - pages/ (Page Object classes)
    - locators/ (selectors per page)
  - api/
    - clients/ (API request context)
    - services/ (endpoint wrappers)
    - payloads/ (request bodies)
    - headers/ (default headers)
    - validators/ (response validators; currently unused)
  - reports/
    - generateHtmlReport.js
  - utils/
    - csvWriter.ts

- config/
  - env.ts
  - urls.ts

- reports/ (generated cucumber json/html)
- output/ (generated CSV extraction outputs)

4) Execution model
------------------
NPM scripts in package.json drive everything:
- cucumber:run (core runner)
- test:ui
- test:api
- test:address
- test:latest
- test:smoke
- test:regression

Each tagged test script runs Cucumber and then generates HTML report via posttest script.

5) Cucumber + runtime wiring
----------------------------
- cucumber.js loads:
  - ts-node/register
  - all hooks in src/hooks/**/*.ts
  - all steps in src/step-definitions/**/*.ts
  - feature files from features/**/*.feature
- Produces:
  - progress output
  - reports/cucumber-report.json
  - reports/cucumber-report.html

6) UI flow internals
--------------------
- Hook for @ui in src/hooks/common.hooks.ts:
  - Launches Chromium (headful, slowMo)
  - Creates browser context/page
  - Adds init script overriding navigator.webdriver
  - Initializes page objects
  - Closes page/context/browser after scenario
  - Captures screenshot attachment on failed steps

- Step definitions in src/step-definitions/ui/checkout-e2e.steps.ts orchestrate:
  - CheckoutPage
  - VehiclePage
  - PlanPage
  - QuotePage
  - PaymentPage

- POM + locator separation:
  - Pages implement behavior and assertions
  - Locator classes encapsulate selectors (role-based + id/placeholder based)

7) API flow internals
---------------------
- ApiClient creates Playwright APIRequestContext with baseURL from config/urls.ts
- Address validation service calls:
  POST /api/esp/address/validate
- Vehicle extraction services call hierarchical GET endpoints:
  - /api/esp/{vehicleType}/years
  - /api/esp/{vehicleType}/year/{year}
  - /api/esp/{vehicleType}/year/{year}/manufacturer/{manufacturer}
  - /api/esp/{vehicleType}/year/{year}/manufacturer/{manufacturer}/model/{model}
  - /api/esp/{vehicleType}/year/{year}/manufacturer/{manufacturer}/model/{model}/trim/{trim}

- addressValidate.steps.ts checks:
  - success case
  - URL-encoded address decoding behavior
  - required field failures (address1, zipCode, country)
  - invalid zip format failure

- vehicleYearFullExtract.steps.ts:
  - Iterates through returned hierarchy
  - Writes master CSV rows
  - Writes failure CSV rows with reason, curl command, raw response snippets
  - Uses Promise.all for per-trim styles requests in autos flow

11) End-to-end pattern summary
------------------------------
The framework follows a clear layered design:
Feature files -> Step definitions -> Page objects / API services -> Utilities (reports, CSV)

This keeps test intent readable (BDD) while isolating technical details in reusable classes.
